# PySyMex Architectural Refactoring: AST to Pure-IR Migration Plan

## 1. Executive Summary & The AST Bottleneck
In "first-generation" symbolic executors, it is common to tightly couple the execution loop to backend ASTs (either the Python interpreter's `ast` module or the SMT solver's C++ ASTs like `z3.ExprRef`). While easy to implement initially, this design inherently limits scalability.

In PySyMex v1, bleeding SMT ASTs into the Virtual Machine loop created catastrophic bottlenecks:
*   **Thread-Safety Collisions:** Z3 Contexts are tied to C++ thread-local storage, forcing the introduction of global locks during asynchronous solving.
*   **Serialization/Hashing Overhead:** Calling `hash()` or `str()` on C++ AST objects in the hot loop forces slow FFI (Foreign Function Interface) calls, decimating paths-per-second throughput.
*   **Analysis Misalignment:** Static analysis passes relying on `ast.parse()` lack the runtime context of the operand stack, jump offsets, and local variables inherently tracked by the Bytecode VM.

**The Solution: The Pure-IR VM.** 
PySyMex v2 must transition entirely to a Bytecode CFG and Custom Intermediate Representation (IR). The VM will only manipulate lightweight, immutable Python dataclasses. Translation to SMT logic (Z3, cvc5, etc.) will occur strictly inside the isolated worker threads at the exact moment of constraint solving.

---

## 2. The Structural Perimeter: Where Python AST MUST Be Preserved
While we are stripping AST out of the core VM, certain security and mathematical boundaries fundamentally require parsing raw source code. Python `ast` will be strictly quarantined to the following subsystems:

1.  **Sandbox Defense (`sandbox/isolation/harness.py`)**
    *   **Role:** Statically vets untrusted code *before* bytecode compilation.
    *   **Reason:** The sandbox uses `ast.NodeVisitor` to catch forbidden nodes (e.g., `Call` to `__import__('os')`) to prevent arbitrary execution before the VM even spins up.

2.  **Formal Contracts (`analysis/contracts/compiler.py` & `contracts/quantifiers/core.py`)**
    *   **Role:** Converts user-defined decorators (e.g., `@requires(lambda x: x > 0)`) into formal constraints.
    *   **Reason:** Translating inline Python lambda syntax into mathematical logic requires parsing the source AST.

3.  **Exploit Reproduction (`reporting/reproduction.py`)**
    *   **Role:** Synthesizes human-readable Python exploit scripts from raw constraints.
    *   **Reason:** `ast` and `ast.unparse` are required to cleanly format and generate the synthesized crash harness.

## 3. The Hit List: Subsystems Slated for Pure-IR Refactoring

Based on comprehensive codebase analysis, the following subsystems currently misuse ASTs and are slated for immediate replacement.

### Target A: The Z3 SMT AST Bleed (The Core Speed Killer)
**Location:** `pysymex/core/types/*` (e.g., `scalars.py`, `numeric.py`, `containers.py`, `symbolic_containers.py`)
*   **Current State:** Classes like `SymbolicValue` construct `z3` objects inline during operations (e.g., `return SymbolicValue(z3.ArithRef(self.expr + other.expr))`).
*   **The Impact:** This forces the VM's main thread to invoke C++ allocation, lock global Z3 Contexts, and perform heavy SMT operations just to execute a simple `BINARY_ADD` bytecode.
*   **Refactor Plan:**
    1.  Design `pysymex.core.ir`: A lightweight, immutable, frozen dataclass hierarchy (`IR.Add`, `IR.Eq`, `IR.Var`).
    2.  Rewrite `SymbolicType` subclasses to return Pure IR nodes instead of Z3 references.
    3.  Build an `IR_to_Z3` compiler strictly inside `AsyncMUSWorker` (or equivalent solver layer).

### Target B: AST-Based Dead Code Analysis
**Location:** `pysymex/analysis/dead_code/core.py`
*   **Current State:** Parses source strings via `ast.parse()` and walks the tree to find unused imports or unreachable `if` blocks.
*   **The Impact:** Extremely slow, redundant parsing that ignores the VM's existing control-flow knowledge.
*   **Refactor Plan:** Transition dead code analysis to consume the Bytecode Control Flow Graph (CFG) generated by `dis.Bytecode`. Dead code is mathematically defined as any sequence of instruction offsets with zero inbound jump edges. This turns an $O(\text{parse})$ operation into a microsecond $O(N)$ graph traversal.

### Target C: AST-Based Exception Routing
**Location:** `pysymex/analysis/exceptions/analyzer.py` & `handler.py`
*   **Current State:** Uses massive `ast.NodeVisitor` classes to manually trace `Try`, `ExceptHandler`, and `Raise` blocks to map exception flows and classify intent.
*   **The Impact:** Prone to line-number misalignment and extremely heavy.
*   **Refactor Plan:** Python 3.11+ bytecode explicitly defines exception routing in the `exception_entries` table of the compiled code object. The Bytecode VM natively knows exactly where exceptions branch. We will drop the AST visitor entirely and reconstruct exception logic purely by traversing the `dis.Bytecode().exception_entries` table during CFG construction.

### Target D: AST-Based Specialized Security Passes
**Location:** `pysymex/analysis/specialized/strings.py` (SQLi, Path Traversal) & `none.py`
*   **Current State:** Uses `ast.NodeVisitor` to hunt for `JoinedStr` (f-strings) or `ast.Compare` (`is None`).
*   **The Impact:** Duplicates the VM's execution logic.
*   **Refactor Plan:** Integrate these checks directly into the `OpcodeDispatcher`. 
    *   For `None` checks: Tag variables natively when the VM executes the `IS_OP` or `COMPARE_OP` bytecode.
    *   For Strings/SQLi: Track `BINARY_ADD` or `FORMAT_VALUE` bytecodes on tainted variables using the existing `TaintTracker` engine.

## 4. Phase 7: The Execution Roadmap

To achieve the Pure-IR architecture without destabilizing the engine, we will execute the transition iteratively:

1.  **Step 7.1: The IR Definition** 
    *   Implement `pysymex/core/ir.py` defining the pure Python expression tree.
2.  **Step 7.2: The SMT Lowering Compiler**
    *   Implement the visitor pattern to translate `IR` -> `z3.ExprRef` safely inside isolated Contexts.
3.  **Step 7.3: Decoupling Types**
    *   Surgically swap `scalars.py` and `containers.py` to emit IR instead of Z3 objects. 
    *   Validate via `tests/unit/core/types`.
4.  **Step 7.4: Bytecode Analysis Migration**
    *   Rewrite `DeadCode` and `ExceptionAnalyzer` to ingest `dis.Instruction` and CFG edges.
5.  **Step 7.5: Cleanup & Deletion**
    *   Eradicate legacy AST visitors and remove global `import z3` from the core type system.

**End State:** A VM that processes 10,000+ paths/second completely decoupled from SMT C++ locking, ready to integrate arbitrary backend solvers (cvc5, Bitwuzla) seamlessly.
